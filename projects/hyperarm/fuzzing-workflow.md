# HyperArm LibAFL Nesting Fuzzing Workflow

This document explains the
`nvirsh-aarch64-libafl-nesting-fuzzing` workflow in
`projects/hyperarm/morpheus.yaml`.

The workflow fuzzes a nested AArch64 VM setup:

```text
host LibAFL qemu_nesting
  -> L1 Debian arm64 VM from nvirsh overlay.qcow2
    -> /root/libafl_nesting_stub runs as L1 init
      -> /root/launch-l2.sh starts nested L2 QEMU
        -> L2 Buildroot kernel and initramfs run under traced QEMU
```

## Layers

The workflow has three layers.

`host`

- Runs Morpheus workflow steps.
- Builds or reuses LibAFL, QEMU, nqc2, and nvirsh artifacts.
- Runs the host-side LibAFL fuzzer binary, `qemu_nesting`.
- Collects the L2 trace and turns it into guest kernel coverage.

`L1`

- A Debian 12 arm64 cloud image prepared by nvirsh.
- Booted by host QEMU with virtualization enabled.
- Contains the nested QEMU build, nqc2 plugin, Buildroot kernel image,
  Buildroot initramfs, and LibAFL guest stub.
- During fuzzing, its kernel starts `/root/libafl_nesting_stub` as `init`.

`L2`

- A Buildroot kernel and initramfs launched by QEMU inside L1.
- This is the guest kernel whose coverage is postprocessed.

## Workflow Steps

The workflow in `projects/hyperarm/morpheus.yaml` performs these steps:

1. `libafl_fetch`, `libafl_patch`, `libafl_build`

   Fetches LibAFL, overlays the `libafl_nesting` crate and
   `qemu_nesting` fuzzer, then builds:

   - `qemu_nesting`
   - `libafl_nesting_stub`
   - the patched `qemu-libafl-bridge`

2. `qemu_fetch`, `qemu_patch`, `qemu_build`

   Builds the host AArch64 QEMU used to boot L1 during provisioning.

3. `nqc2_fetch`, `nqc2_build`

   Builds nqc2 and the AArch64 QEMU plugin used inside L1.

4. `nvirsh_fetch`, `nvirsh_build`

   Prepares the Debian L1 overlay image and injects all fuzzing payloads.

5. `libafl_exec`

   Runs `qemu_nesting` against the prepared L1 overlay for the configured
   fuzzing duration.

6. `nvirsh_inspect`

   Extracts trace artifacts from the prepared nvirsh build tree after fuzzing.

7. `nqc2_postprocess`, `nqc2_genhtml`

   Converts the nqc2 trace into LCOV and HTML coverage for the L2 kernel.

## L1 Provisioning

L1 provisioning happens during `nvirsh_build`.

The nvirsh profile is:

```text
tools/nvirsh/profiles/qemu-debian-arm/profile.json
```

The L0/L1 scripts are generated by:

```text
tools/nvirsh/scripts/l0-provision.sh
tools/nvirsh/scripts/l1-provision.sh
tools/nvirsh/scripts/build.sh
```

Provisioning does the following:

1. Downloads or reuses the Debian 12 arm64 cloud image as
   `base-image.qcow2`.
2. Creates `overlay.qcow2` backed by `base-image.qcow2`.
3. Creates a cloud-init seed image with SSH access.
4. Boots L1 normally with systemd and SSH.
5. Waits for cloud-init to finish.
6. Copies these files into L1:

   ```text
   /root/libafl_nesting_stub
   /root/launch-l2.sh
   /root/nvirsh-images/Image
   /root/nvirsh-images/rootfs.cpio.gz
   /root/morpheus-nqc2/lib/nqc2/nqc2-plugin.so
   /root/morpheus-qemu-src/
   ```

7. Builds nested QEMU inside L1:

   ```text
   /root/morpheus-qemu/bin/qemu-system-aarch64
   ```

8. Updates L1 GRUB so future L1 boots use:

   ```text
   init=/root/libafl_nesting_stub norandmaps rw
   ```

9. Powers off L1 and records the prepared state in:

   ```text
   $CACHE/tools/nvirsh/builds/qemu-debian-arm64/install/state.json
   ```

After this phase, `overlay.qcow2` is a prepared L1 fuzzing image.

## Fuzzing Execution

Fuzzing happens during `libafl_exec`.

The host command assembled by `tools/libafl/scripts/exec.sh` is equivalent to:

```bash
CACHE="$HOME/.cache/morpheus/hyperarm"
FW="$CACHE/tools/qemu/builds/qemu-8.2.7-aarch64-softmmu/install/share/qemu"
DISK="$CACHE/tools/nvirsh/builds/qemu-debian-arm64/build/l0/overlay.qcow2"
QEMU_DATA="$CACHE/tools/libafl/builds/libafl-main/build/qemu-libafl-bridge"

STUB="$CACHE/tools/libafl/builds/libafl-main/install/bin/libafl_nesting_stub" \
"$CACHE/tools/libafl/builds/libafl-main/install/bin/qemu_nesting" \
  -machine virt,virtualization=on,gic-version=3 \
  -cpu cortex-a57 \
  -m 8192 \
  -smp 4 \
  -nographic \
  -bios "$FW/edk2-aarch64-code.fd" \
  -drive "file=$DISK,if=virtio,format=qcow2" \
  -L "$QEMU_DATA/build/qemu-bundle/usr/local/share/qemu"
```

`qemu_nesting` boots L1 from `overlay.qcow2`. Because GRUB was modified during
provisioning, L1 does not boot normal userspace during fuzzing. Instead, the
L1 kernel starts:

```text
/root/libafl_nesting_stub
```

as PID 1.

`qemu_nesting` is still the host-side LibAFL fuzzer. It does not replace the
guest stub. The guest stub and `libvharness` provide the control channel back
to the host-side fuzzer.

The runtime relationship is:

```text
qemu_nesting
  -> LibAFL scheduler, mutator, executor, and feedback
  -> libafl_qemu system-mode emulator
  -> QEMU edge coverage module
  -> libvharness command parser
    -> L1 libafl_nesting_stub
      -> libvharness/lqemu custom instruction calls
      -> /root/launch-l2.sh
```

In short:

```text
qemu_nesting owns fuzzing orchestration.
libafl_nesting_stub owns the L1 guest harness.
libvharness is the guest/host command ABI between them.
```

## Snapshot Boundary

L1 is not rebooted for every fuzz input.

Within one `qemu_nesting` launcher attempt:

```text
boot L1 once
  -> L1 kernel starts libafl_nesting_stub
    -> LibAFL reaches the start boundary
      -> snapshot/restart point is established
      -> each fuzz input runs from that point
```

If the `qemu_nesting` launcher crashes or is restarted, L1 boots again and the
snapshot state is recreated.

The qcow2 overlay also contains an internal QEMU VM snapshot:

```text
__libafl_qemu_snapshot_0
```

List it with:

```bash
qemu-img snapshot -l \
  "$CACHE/tools/nvirsh/builds/qemu-debian-arm64/build/l0/overlay.qcow2"
```

The practical start and end boundaries are not a symbol named `BREAKPOINT`.
This implementation uses the LibAFL QEMU command ABI in the L1 stub:

```text
projects/hyperarm/workspace/tools/libafl/patches/overlay/crates/
  libafl_nesting/c_src/libafl_nesting_stub.c
```

The relevant stub loop is:

```c
size_t len = (size_t)libafl_qemu_start_virt(FUZZ_INPUT, INPUT_LEN);
bool ok = write_input_snapshot(FUZZ_INPUT, len) && launch_l2(FUZZ_INPUT);
libafl_qemu_end(ok ? LIBAFL_QEMU_END_OK : LIBAFL_QEMU_END_CRASH);
```

At `libafl_qemu_start_virt`, the host-side fuzzer provides the next serialized
`ScenarioInput` into the guest buffer:

```text
FUZZ_INPUT
```

`libafl_qemu_start_virt` itself is implemented by the generated `libvharness`
LQEMU shim:

```text
$CACHE/tools/libafl/builds/libafl-main/build/target-host/debug/
  vharness_out/src/lqemu.c
```

The guest-side implementation is a thin custom-instruction call:

```c
return _lqemu_custom_insn_call2(
    LIBAFL_QEMU_COMMAND_START_VIRT,
    (vword)buf_vaddr,
    max_len
);
```

The host-side parser for that command is in:

```text
$CACHE/tools/libafl/builds/libafl-main/source/crates/libafl_qemu/
  src/command/lqemu/parser.rs
```

The command handler saves the initial snapshot and records the input buffer
location in:

```text
$CACHE/tools/libafl/builds/libafl-main/source/crates/libafl_qemu/
  src/command/lqemu/mod.rs
```

The built stub currently has:

```text
main       0x400500
FUZZ_INPUT 0x4a1990
```

## Input Path

The fuzz input reaches nested QEMU in two stages.

First, host LibAFL writes the serialized `ScenarioInput` into L1 guest memory.

The L1 stub passes the guest virtual address and maximum length to the host:

```c
libafl_qemu_start_virt(FUZZ_INPUT, INPUT_LEN)
```

The host parser reads those arguments from the guest register ABI and records
the input location:

```text
$CACHE/tools/libafl/builds/libafl-main/source/crates/libafl_qemu/
  src/command/lqemu/parser.rs
```

The input setter later writes each input into that recorded guest memory
location:

```text
$CACHE/tools/libafl/builds/libafl-main/source/crates/libafl_qemu/
  src/emu/drivers/lqemu.rs
```

The relevant operation is:

```rust
input_location.write(input.target_bytes().as_slice())
```

Then the L1 stub consumes `FUZZ_INPUT`.

Second, the L1 stub converts that input into nested QEMU inputs:

1. It writes the raw bytes to:

   ```text
   /root/morpheus-qemu-input.bin
   ```

2. It starts `/root/launch-l2.sh` with:

   ```text
   MORPHEUS_QEMU_INPUT_PATH=/root/morpheus-qemu-input.bin
   MORPHEUS_QEMU_INJECT_VIRQ_PERIOD_MS=<derived from bytes 1..2>
   MORPHEUS_QEMU_INJECT_VIRQ=<derived from byte 0, when nonzero>
   ```

3. The patched nested QEMU reads those environment variables.

The virtio-mmio patch reads the input file in:

```text
$CACHE/tools/qemu/src/qemu-8.2.7/hw/virtio/virtio-mmio.c
```

It uses `MORPHEUS_QEMU_INPUT_PATH`, reads the file, and XOR-mixes bytes into
selected virtio-net MMIO reads.

The GICv3 patch reads interrupt injection settings in:

```text
$CACHE/tools/qemu/src/qemu-8.2.7/hw/intc/arm_gicv3_cpuif.c
```

It uses `MORPHEUS_QEMU_INJECT_VIRQ` and
`MORPHEUS_QEMU_INJECT_VIRQ_PERIOD_MS` to inject virtual interrupts.

The full path is:

```text
ScenarioInput
  -> LibAFL target bytes
  -> LqemuInputSetter
  -> L1 FUZZ_INPUT buffer
  -> /root/morpheus-qemu-input.bin
  -> MORPHEUS_QEMU_* environment
  -> patched nested QEMU virtio-mmio and GICv3 behavior
```

## What Happens Inside L1 Per Input

For each fuzz input, the L1 stub:

1. Receives up to 512 bytes in `FUZZ_INPUT`.
2. Writes the raw input to:

   ```text
   /root/morpheus-qemu-input.bin
   ```

3. Derives runtime knobs from the input, including:

   ```text
   MORPHEUS_QEMU_INPUT_PATH
   MORPHEUS_QEMU_INJECT_VIRQ_PERIOD_MS
   MORPHEUS_QEMU_INJECT_VIRQ
   ```

4. Runs:

   ```text
   /root/launch-l2.sh
   ```

5. Lets L2 run for the input-derived time window.
6. Reports `OK` or `CRASH` to host LibAFL with `libafl_qemu_end`.

`/root/launch-l2.sh` starts nested QEMU inside L1:

```bash
/root/morpheus-qemu/bin/qemu-system-aarch64 \
  -trace events=/root/morpheus-qemu-trace-events.txt,file=/dev/stderr \
  -plugin /root/morpheus-nqc2/lib/nqc2/nqc2-plugin.so,trace=TRACE \
  -machine virt,virtualization=on,gic-version=3 \
  -accel tcg \
  -cpu cortex-a57 \
  -m 1024 \
  -nographic \
  -kernel /root/nvirsh-images/Image \
  -initrd /root/nvirsh-images/rootfs.cpio.gz \
  -netdev user,id=net0 \
  -device virtio-net-device,netdev=net0 \
  -append "console=ttyAMA0"
```

The L1 script writes this QEMU trace event file:

```text
virtio_mmio_fuzz_read
```

## LibAFL Feedback

LibAFL's fuzzing feedback is collected by host-side QEMU edge
instrumentation, not by the final nqc2 LCOV report.

The host fuzzer configures an edge map observer in:

```text
projects/hyperarm/workspace/tools/libafl/patches/overlay/
  fuzzers/full_system/qemu_nesting/src/fuzzer_breakpoint.rs
```

The relevant setup is:

```rust
HitcountsMapObserver::new(VariableMapObserver::from_mut_slice(
    "edges",
    OwnedMutSlice::from_raw_parts_mut(
        edges_map_mut_ptr(),
        EDGES_MAP_DEFAULT_SIZE,
    ),
    &raw mut MAX_EDGES_FOUND,
))
```

That observer is passed to the LibAFL QEMU edge module:

```rust
StdEdgeCoverageModule::builder()
    .map_observer(edges_observer.as_mut())
    .build()?
```

The fuzzer then treats increased edge-map coverage as interesting with:

```rust
MaxMapFeedback::new(&edges_observer)
```

The active feedback tuple is:

```rust
feedback_or!(
    MaxMapFeedback::new(&edges_observer),
    TimeFeedback::new(&time_observer)
)
```

Crash and timeout are objectives:

```rust
feedback_or_fast!(CrashFeedback::new(), TimeoutFeedback::new())
```

In short:

```text
QEMU TCG executes guest code
  -> LibAFL QEMU edge module updates EDGES_MAP
    -> HitcountsMapObserver reads the map
      -> MaxMapFeedback detects new or increased edges
        -> input is kept/mutated if it improves feedback
```

This feedback guides mutation during fuzzing. It is separate from the
post-run guest kernel coverage generated from nqc2 traces.

## Throughput Notes

Current throughput is expected to be low.

Each LibAFL execution is not a simple in-process function call. One execution
does the following:

```text
restore L1 snapshot
  -> write input into L1 memory
  -> L1 stub writes /root/morpheus-qemu-input.bin
  -> L1 stub forks /bin/bash /root/launch-l2.sh
  -> launch-l2 starts nested qemu-system-aarch64
  -> L2 Buildroot kernel/initramfs boot under TCG
  -> patched QEMU and nqc2 tracing run
  -> stub reports OK/CRASH/TIMEOUT
```

The current setup avoids rebooting L1 for every input, but it still starts L2
QEMU for every input. L2 also runs with:

```text
-accel tcg
```

That means the fuzz target is nested emulation plus process startup and kernel
boot work per input.

The LibAFL monitor's `exec/sec` value is wall-clock executions per second for
that full path. Values well below 1 exec/sec are therefore plausible.

Main ways to improve throughput:

- Snapshot after L2 is already booted, not only after L1 stub startup.
- Keep the nested QEMU process alive across inputs.
- Avoid per-input filesystem writes in L1.
- Reduce trace/log volume during fuzzing.
- Shorten the per-input run window and executor timeout.

## Coverage

After fuzzing, `nvirsh_inspect` and nqc2 postprocessing extract and convert the
guest trace.

The coverage outputs are written under the `libafl_exec` runtime directory for
the workflow run:

```text
projects/hyperarm/workspace/runs/<run-id>/steps/libafl_exec/runtime/
  guest-kernel-coverage.info
  guest-kernel-coverage-html/
```

Summarize LCOV with:

```bash
lcov --summary \
  projects/hyperarm/workspace/runs/<run-id>/steps/libafl_exec/runtime/\
guest-kernel-coverage.info
```

## Useful Commands

Run through the Morpheus workflow:

```bash
./bin/morpheus --config projects/hyperarm/morpheus.yaml \
  workflow run nvirsh-aarch64-libafl-nesting-fuzzing --json
```

Resume the checked-out run from `libafl_exec` with a chosen duration:

```bash
projects/hyperarm/run-libafl-fuzzing.sh --minutes 10
projects/hyperarm/run-libafl-fuzzing.sh --hours 1
```

Run the extracted raw command without Morpheus:

```bash
projects/hyperarm/run-libafl-fuzzing-raw.sh --minutes 10
projects/hyperarm/run-libafl-fuzzing-raw.sh --hours 1
```

Inspect the prepared overlay:

```bash
qemu-img info \
  "$CACHE/tools/nvirsh/builds/qemu-debian-arm64/build/l0/overlay.qcow2"
```

List the internal VM snapshot:

```bash
qemu-img snapshot -l \
  "$CACHE/tools/nvirsh/builds/qemu-debian-arm64/build/l0/overlay.qcow2"
```
